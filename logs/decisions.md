# Decision Journal

### 2026-04-06 — Merged /repo-dd-deep into /repo-dd as depth levels
- **Context:** Built `/repo-dd-deep` as a standalone command for operational health review. Patrik asked whether the two commands could be combined for a "comprehensive audit."
- **Decision:** Extended `/repo-dd` with `$ARGUMENTS` depth control: (empty) = factual audit, "deep" = audit + judgment, "full" = audit + judgment + pipeline testing. Two separate report files preserve evidence/interpretation separation.
- **Rationale:** Eliminates two-session friction and the freshness dependency (deep review needed a recent audit as input). The separation principle is maintained at the report level, not the command level.
- **Alternatives considered:** (A) Combined command with separate reports (chosen), (B) Single combined report (rejected — blurs facts/judgment separation required by CLAUDE.md design principles).

### 2026-04-06 — Shared commands distributed via --add-dir, not copied into projects
- **Context:** Creating `/wrap-session` for ai-resources raised the question of how to make it available to all projects.
- **Decision:** Shared commands live in `ai-resources/.claude/commands/` and are available to all connected projects via `--add-dir`. Projects that need a specialized version override locally (local commands take precedence).
- **Rationale:** Single source of truth — updates propagate automatically. No sync burden. Research-workflow projects already override with their pipeline-specific version, proving the override pattern works.
- **Alternatives considered:** (a) Copy into each project at deploy time via `/deploy-workflow` — creates drift. (b) User-level commands in `~/.claude/commands/` — less portable, not version-controlled with the repo.

### 2026-04-06 — /repo-dd and /repo-review are separate commands
- **Context:** After running the first due diligence audit, Patrik wanted a command that also answers operational questions (feature criticality, context management health, friction synthesis, functional pipeline testing). These are judgment-based questions that don't fit the questionnaire's facts-only design.
- **Decision:** Keep `/repo-dd` as a structural audit (facts, no interpretation, comparable across runs). Build `/repo-review` separately for operational health assessment (judgment, recommendations, reads friction logs and tests pipelines).
- **Rationale:** Mixing factual audit with interpretive assessment would break the questionnaire's repeatability. A structural audit that says "this file is stale" is different from an operational review that says "this feature is your highest priority to fix." Different inputs, different outputs, different cadence.
- **Alternatives considered:** (a) Expand `/repo-dd` to include operational questions — rejected because it introduces judgment into a facts-only tool. (b) One command with two modes — adds complexity without benefit.

### 2026-04-06 — AUTO-FIX triage defaults to OPERATOR when ambiguous
- **Context:** The /repo-dd command includes a triage step that categorizes audit findings for fixing. The risk is Claude Code being too aggressive about what it auto-fixes.
- **Decision:** AUTO-FIX is strictly conservative: only unambiguous, single-file, no-cascade fixes. Everything else defaults to OPERATOR. The triage instruction explicitly says "when in doubt, classify as OPERATOR."
- **Rationale:** A conservative auto-fix list is more trustworthy than an ambitious one. Claude Code may misjudge whether a naming inconsistency is a simple rename or a cascading change. Better to surface more items for operator decision than to silently break references.
- **Alternatives considered:** (a) More permissive auto-fix with rollback — adds complexity, harder to verify. (b) No auto-fix at all — loses the convenience of fixing genuinely trivial items.

### 2026-04-06 — Exclusion-list approach for feature enrichment in project deployment
- **Context:** New features added to ai-resources (commands, agents, hooks) weren't reaching newly deployed projects because `/deploy-workflow` copies from a stale template snapshot.
- **Decision:** Use an exclusion list (repo-specific items blocked) rather than an inclusion list (shared items allowed). New features flow to projects by default.
- **Rationale:** Patrik constantly adds new features. An inclusion list requires manual updates every time. An exclusion list only needs updating when a new repo-specific tool is created, which is rare. The 9 excluded commands, pipeline agents, and 2 hooks are clearly ai-resources-only.
- **Alternatives considered:** (a) Inclusion list — rejected, requires constant maintenance. (b) Keep template manually updated — rejected, that's the discipline problem this fix solves. (c) Symlinks instead of copies — rejected, projects are standalone repos that may diverge intentionally.

### 2026-04-06 — Shared command drift check over dedicated sync command
- **Context:** Improvements made to shared commands in project workspaces (e.g., buy-side adding coaching data to wrap-session) don't automatically flow back to ai-resources. No mechanism existed to catch this drift.
- **Decision:** Add a lightweight drift check step to `/wrap-session` rather than building a dedicated `/sync-command` skill.
- **Rationale:** Piggybacks on an existing routine (session wrap) so it costs zero extra process. Only asks the question — doesn't force action. If this proves insufficient, a dedicated command can be built later.
- **Alternatives considered:** (a) Manual habit — rejected, relies on memory. (b) Dedicated `/sync-command` — deferred, over-engineering for a problem that may be fully solved by a prompt. (c) Automated diffing — rejected, too complex for the current volume of shared commands.

### 2026-04-11 — Bright-line check 1 exemption for AI prose decontamination
- **Context:** The produce-prose pipeline's bright-line rule requires operator approval for multi-paragraph changes. The new Phase 5c (AI prose decontamination) operates across the entire document by design — every invocation makes multi-paragraph changes. Requiring approval on check 1 would make every invocation a manual gate, defeating the automatic flow design.
- **Decision:** Exempt Phase 5c from bright-line check 1 (multi-paragraph scope). Checks 2 (analytical claim alteration) and 3 (sourced statement modification) remain active and trigger a PAUSE if violated.
- **Rationale:** The bright-line rule protects analytical content from unreviewed changes. Decontamination explicitly does not change analytical content — it changes surface-level voice patterns. The exemption preserves the rule's intent while allowing the pass to function. This is the first exemption to the bright-line rule in the workflow.
- **Alternatives considered:** (a) Full bright-line compliance — rejected, creates a mandatory gate on every invocation. (b) Raising the threshold (e.g., exempt only if <50 changes) — rejected, arbitrary threshold that still gates most invocations.

### 2026-04-13 — Codex second-opinion auditor: independent executor pattern, mechanical wrapper, strict ordering
- **Context:** Operator raised the blind-spot risk that every audit and QC pass in this workspace is executed by Claude (Opus 4.6). Subagent isolation gives fresh context but not a fresh model — two Claude instances evaluating the same artifact share training data and failure modes, so some gaps stay invisible across arbitrarily many review passes. Question: can a different model family serve as a genuine second opinion?
- **Decision:** Viability investigation concluded positively with a specific pattern. (1) Codex CLI (`codex exec`) runs the workspace's audit *framework files* natively — not Claude's subagent-dependent command wrappers. Framework files (`audits/questionnaire.md`, `skills/ai-resource-builder/references/evaluation-framework.md`) are model-agnostic; command files are not. (2) Strict ordering rule: Claude produces its audit first, Codex produces its audit second, never reverse. (3) Wrapper prompt from Claude to Codex must be mechanical — paths + schema only, no editorial framing — to preserve independence. (4) Park as inbox brief this session; pilot is a single command `/codex-dd` using the existing questionnaire, one real run, then decide whether to expand.
- **Rationale:** `codex exec` is a fresh-context agentic loop structurally equivalent to a Claude subagent, with stronger process-level isolation (no shared conversation state at all). The QC Independence Rule is satisfied by construction. Running against framework files rather than command wrappers avoids coupling to Claude Code's Agent-tool execution model, which Codex cannot replicate. The ordering rule matters because the value depends on each model forming its view without seeing the other's; reverse ordering silently breaks the pattern. The mechanical wrapper prompt matters because Claude's natural impulse to "help" by summarizing prior findings is exactly what contaminates Codex's independent evaluation. Starting with one command lets a single real run reveal format-divergence issues and value signal before committing to porting all six subagent-dependent audit commands.
- **Alternatives considered:** (a) Codex reviews Claude's report as a second-reviewer (rejected — two models reading the same summary share more signal than two models reading the same source material independently, and it introduces a cognitive anchor that defeats the independence goal). (b) Extend the existing `.claude/agents/execution-agent.md` HTTP pattern for GPT-5/Perplexity to cover Codex (rejected — Codex CLI is a multi-turn agentic process with its own tools, not a one-shot request/response call; forcing it through the HTTP pattern loses the value). (c) Build the `/codex-dd` pilot this session (rejected — operator preferred a conversational viability memo parked as a draft for future pickup, preserving optionality). (d) Port all six subagent-dependent audit commands at once (rejected — format divergence and value signal are unknown; a single pilot produces those answers cheaply). (e) Run Codex first, then Claude (rejected — Claude seeing Codex output first contaminates Claude's independence, which is a one-way door).

### 2026-04-13 — Reversed "additionalDirectories intentionally omitted" decision in /new-project
- **Context:** The `/new-project` pipeline's post-pipeline enrichment step previously kept the canonical permissions block scoped to `allow`/`deny` arrays only, with an explicit comment that `additionalDirectories` was "intentionally omitted because it is a user-level absolute path". In practice this meant newly scaffolded projects couldn't read through symlinks into `ai-resources/` — skills (under `ai-resources/skills/`) were invisible to Claude Code's per-project sandbox, and even the symlinked commands under the project's `.claude/commands/` failed to load because Claude Code cannot follow a symlink into a directory it hasn't been granted. The operator's global `~/.claude/settings.json` `additionalDirectories` entry for the workspace root was the only thing keeping things working, and it had proven unreliable in cascading to per-project sessions. The operator reported repeatedly hitting this with new-project-pipeline output.
- **Decision:** Reverse the earlier decision. `permissions.additionalDirectories` is now added per-project at post-pipeline enrichment time via a new numbered step 3 that walks upward from the project directory to locate the ancestor containing `ai-resources/`, then runs a separate idempotent `jq` merge on `.permissions.additionalDirectories`. The step is independent from step 2's canonical-permissions merge so that projects with pre-existing `permissions.allow` arrays (which skip the canonical merge via the existing predicate) still receive the grant. Retrofit applies to all 5 existing projects. `/repo-dd` gets Q3.5 + Q3.6 so any future regression is caught on audit.
- **Rationale:** Per-project entries are version-controlled, travel with the project (anyone else opening the project gets the grant without touching user-level settings), defend-in-depth against global-setting drift, and are auditable via the new questionnaire items. The absolute-path concern that originally motivated the exclusion is addressed by computing the path dynamically at enrichment time via an upward walk instead of hardcoding it into a canonical template. Skills under `ai-resources/skills/` are the load-bearing case: `auto-sync-shared.sh` only syncs commands and agents, so there is no symlink fallback for skills — `additionalDirectories` is the only mechanism that exposes them to a project session.
- **Alternatives considered:** (a) Launch-time `--add-dir ai-resources` via a README or launch script (rejected — relies on operator discipline, not version-controlled, easy to forget). (b) Rely solely on the operator's global `~/.claude/settings.json` entry (rejected — operator-global setting has proven unreliable in this workspace, and it isn't self-contained per project). (c) Relative path like `"../../ai-resources"` in the field (rejected — Claude Code resolves `additionalDirectories` relative to session CWD which varies by launch method, so relative paths can silently misresolve; the operator's global uses absolute form and we match that). (d) Bake `additionalDirectories` into the canonical CANONICAL_PERMS block in step 2 (rejected — the predicate at step 2 skips the merge entirely for projects that already have `permissions.allow` populated, which would leave those projects ungranted; a separate step 3 decouples the two concerns).

### 2026-04-13 — Commit Rules propagate by explicit copy to every project CLAUDE.md, not inheritance
- **Context:** The workspace-level `CLAUDE.md` at the Axcíon AI workspace root contains a canonical "Commit behavior" section stating "Commit directly. Do not ask for permission." Claude Code walks upward through parent directories when loading CLAUDE.md, so a project session *should* inherit that rule. In practice, this session demonstrated repeatedly that the inheritance path is silent and unreliable — the AI asked the operator for commit permission four times in one session despite the rule being in the parent CLAUDE.md, and the operator eventually escalated ("A MILLION TIMES"). Inheritance evidently does not surface the rule prominently enough to be load-bearing behavior.
- **Decision:** Copy a short `## Commit Rules` section into every project and template CLAUDE.md in the workspace, and add a Post-Pipeline Enrichment step to `/new-project` that creates or appends the same section to any new project's CLAUDE.md. The research-workflow template CLAUDE.md also receives the section so projects deployed via `/deploy-workflow` inherit it. The short form references the canonical workspace-level "Commit behavior" section as the source of truth, so canonical wording drift (if any) still has a single home in the workspace CLAUDE.md, and each project's short form is explicitly a mirror. Applied across 9 existing CLAUDE.md files this session; future projects pick it up automatically via the two pipeline paths.
- **Rationale:** Explicit beats implicit when the rule governs AI behavior that costs the operator real frustration every time it is ignored. Repeating the rule in every project CLAUDE.md guarantees that *whatever* CLAUDE.md loading order Claude Code uses, the rule is present in-context. The cost is a ~8-line duplication across ~10 files and a slight drift risk on the short-form wording, both acceptable. The alternative — debug Claude Code's CLAUDE.md loading order to make the parent rule more reliably visible — is unknowable from outside and fragile. Cost of being explicit is low; cost of guessing wrong is a whole category of operator-facing friction.
- **Alternatives considered:** (a) Only put the rule in the workspace-level CLAUDE.md and rely on inheritance (rejected — empirically failed this session). (b) Put the full canonical rule body in every project CLAUDE.md instead of a short-form reference (rejected — higher drift risk, and the short form is already sufficient because Claude can always re-read the canonical section). (c) Build a hook that injects the rule into every session transcript at load time (rejected — over-engineering for a problem solved by one-time file edits, and hooks run in a different context than CLAUDE.md). (d) Let new projects continue to depend on inheritance and only fix the *existing* projects (rejected — would leave the same hole open for every future project created via `/new-project` or `/deploy-workflow`).
