### 2026-04-18 — R2 Phase 1: split `token-audit-auditor` into two agent files rather than per-call caller-side model override

- **Context:** `/token-audit ai-resources` 2026-04-18 recommendation R2 Phase 1 calls for section-aware model selection on the `token-audit-auditor` subagent — Haiku for mechanical Sections 2/5/6, Opus for judgment Section 4. The first draft plan chose a third, unsanctioned shape — instruct `token-audit.md` to pass `model: haiku` as a per-call Agent-tool override at each Section 2/5/6 launch site. Independent QC pass flagged this as BLOCKING-IF-WRONG: the Task/Agent tool's schema may or may not honor a `model` parameter; if it doesn't, the prose instruction becomes a no-op with no error signal, and the "fix" delivers zero savings while appearing successful.
- **Decision:** Switch to the **split-agent-file** shape. Create a new `token-audit-auditor-mechanical` agent file with `model: haiku` for Sections 2/5/6; narrow the existing `token-audit-auditor` description to Section 4 only (`model: opus` unchanged); update `token-audit.md` Steps 20/32/34 to invoke the mechanical agent and Step 11 existence-check to verify both files exist. - **Rationale:** The split-file shape is declarative (frontmatter-enforced), greppable (`model:` line per file), verifiable by file diff, and sanctioned by the audit text. It does not depend on undocumented tool-schema behavior. The agent body remains identical between the two files because the protocol sections share measurement scaffolding; only the description and model field differ. Future Opus-4.7 pricing shifts change only the Agent Tier Table, not the implementation mechanism.
- **Alternatives considered:** (a) Caller-side model override at each `token-audit.md` launch site — rejected by QC as mechanism-unverified; a future run could silently lose all R2 Phase 1 savings with no error. (b) Conditional logic in the agent body ("if SECTION in (2,5,6) run on Haiku else Opus") — rejected because agent-body prose cannot change the runtime model the harness assigned at subagent launch; the `model:` frontmatter is the sole input the harness reads. ### 2026-04-18 — Plan-schema operator-decision flags live in a Section 4 addendum, not a new section

- **Context:** First real invocation of `/cleanup-worktree` produced a plan with two operator decisions that did not fit the Section 4 hard-gate schema: (a) whether to commit the workflow-template copies of `qc-pass.md` / `refinement-pass.md` as local forks or convert them to symlinks, and (b) whether to commit or delete the already-consumed `inbox/worktree-cleanup-brief.md`. Triage surfaced the first-class alternative of *hoisting* operator-decision flags out of Section 4 entirely into a sibling section, versus keeping them as a *Section 4 addendum* with a clearly-labeled non-hard-gate sub-schema.
- **Decision:** Preserve the skill's 8-section plan schema by restructuring Section 4 into (i) "Hard gates in this plan: NONE" lead paragraph enumerating the classes of irreversible operation that are absent, followed by (ii) a clearly-labeled "Section 4 addendum — Operator-decision flags (non-hard-gate sub-schema)" with its own defined fields: `question`, `paths affected`, `operator options`, `default option`, `abort / revision scope`, `justification`. Each addendum block explicitly declares that it is NOT part of the Section 4 hard-gate inventory schema defined in `execution-protocol.md` § 7. Operator-decision flags therefore live inside Section 4 (as an addendum) rather than as a ninth section or hoisted sibling.
- **Rationale:** The skill's invocation contract (`SKILL.md` → Invocation Contract) requires the eight sections defined in `execution-protocol.md` → "Plan file schema" precisely because two QC subagents and one triage subagent read the plan as a stable contract. Introducing a ninth section unilaterally in the first real invocation would silently change the contract in a way subagents cannot predict. An addendum inside Section 4 with an explicit "NOT part of § 7 schema" label is a smaller, self-documenting extension that preserves the contract while making the non-hard-gate nature unambiguous to a second reader. The cost is mild verbosity in Section 4; the benefit is every plan produced under the unmodified skill will parse the same way.

(c) Embed flags only in Section 8 as "revision-required branches" (rejected — Section 8 is revision history, not a pre-commit gate surface, and subagents would not read it with the right semantics). (d) Skip the schema fidelity entirely and put flags wherever convenient (rejected — was the original plan's path and the first QC pass caught it as BLOCKING).
- **Follow-up:** Logged as `/improve-skill worktree-cleanup-investigator` item #6 in the session plan — if the addendum pattern proves useful across multiple cleanups, consider formalizing it in `execution-protocol.md` → "Plan file schema" so future plans have a defined home for operator-decision flags rather than inventing one. ### 2026-04-17 — ai-prose-decontamination: integrate five new sub-patterns into the existing 4-pass structure, not as a new Pass 5; skill's output-path contract becomes caller-owned
- **Context:** The 2026-04-17 buy-side outside-read feedback on Doc 2 v1.0 surfaced five AI-tell patterns that the `ai-prose-decontamination` skill did not explicitly cover: contrast-template overuse, abstract-noun stacking, pseudo-maxim repetition, pivot closings, and Flagged-Word Registry application. The operator's improvement-log entry named each pattern and proposed candidate placements ("under Pass 1 or as a new Pass" for contrast-template; "under Pass 2 or Pass 4" for pivot closings; etc.). Independently, the Step 4 evaluator subagent flagged the skill's existing "Write Output" directive — "overwrite the input file" — as conflicting with the workspace-level Input File Handling rule (inputs are read-only) and with the operator's versioning convention (new version files rather than in-place overwrites).
- **Decision 1 — Structure:** Integrate five new sub-patterns into the existing 4-pass structure rather than adding a Pass 5. Each new pattern has a functional home inside an existing pass: contrast-template and abstract-noun stacking are vocabulary/ornament failures (Pass 1); Flagged-Word Registry is also a Pass 1 vocabulary authority; pivot closings are deletable-without-loss sentences (Pass 2); pseudo-maxim repetition is a rhythm failure (Pass 4). Sub-patterns are numbered (1a/1b/1c/2a/4a) with first-class H3 headings; Pass 3's existing three sub-patterns renumbered 3a/3b/3c during iteration for symmetry. The skill's frontmatter description retained the 4-pass framing and expanded only the parenthetical coverage list.
- **Decision 2 — Output contract:** The skill's default no longer overwrites the input file. The caller specifies the output path at invocation. When the caller passes the input path as the output (as `produce-prose-draft` Phase 5 does — the command owns the file-versioning contract for the pipeline), in-place overwrite happens explicitly. Standalone invocation defaults to a new versioned path (e.g., `{prose}-decontaminated.md` or `{prose}.v{n+1}.md`), preserving the pre-decontamination file for before/after comparison without going to git. A silent-overwrite-on-any-invocation is prohibited.
- **Rationale (Decision 1):** A new Pass 5 would have collected the five patterns as a grab-bag "control rhetorical habit" pass, weakening Pass 4's rhythm focus and creating category confusion between the new Pass 5 and the existing Pass 2 (which already governs deletable-without-loss sentences like pivot closings). Placing each pattern under its functional parent preserves the skill's existing mental model (four failure modes; passes address them in sequence) while elevating each new pattern to first-class visibility via numbered sub-headings. The change-log format was extended to carry sub-pattern counts, so downstream observability was not lost.
- **Rationale (Decision 2):** The skill was silently overwriting input files, which violated two workspace rules (Input File Handling, Version-file convention) and made pipeline vs. standalone behavior identical when they should differ. Pipelines that own the versioning contract can still overwrite by passing the input path explicitly. Standalone invocation now defaults to a safer behavior that preserves the pre-decontamination state. The caller-controlled output path keeps the skill domain-agnostic without requiring it to know the calling context.
- **Alternatives considered (Decision 1):** (a) Add a new Pass 5 consolidating the five patterns — rejected, weakens category coherence and creates overlap with Pass 2/Pass 4. (b) Add three new passes grouping the patterns differently (one for rhetorical habit, one for vocabulary, one for closings) — rejected, triples the passes for five sub-patterns, poor leverage. (c) Leave the five patterns as inline detection bullets in `produce-prose-draft` Phase 5 and skip the skill update — rejected by the workspace rule that skill-level procedural knowledge belongs in the skill, not in command prose.
- **Alternatives considered (Decision 2):** (a) Keep "overwrite input file" and flag the Input File Handling rule as an exception — rejected, silent exceptions erode the rule. (b) Force all invocations to write to a new path regardless of caller — rejected, pipeline commands legitimately own file versioning and need the option to overwrite. (c) Add a frontmatter flag (`overwrite: true|false`) — rejected, over-engineering; a path parameter already carries this information via the caller's choice.

### 2026-04-18 — Commit-boundary sequencing rule: edit-commit-edit-commit when a single file carries changes for multiple planned commits

- **Context:** The token-audit quick-win plan specified three commits in the buy-side repo: (B) R1 + R8 on settings.json + CLAUDE.md, (C) R4 frontmatter on 7 commands + R4 Model Selection section on CLAUDE.md. Both R8 deletions and R4 Model Selection addition edited the same file (`CLAUDE.md`). When I applied both edits to CLAUDE.md first, then ran `git add CLAUDE.md .claude/settings.json` and committed, the Model Selection section got swept into commit B rather than commit C. `git add` cannot split a single file by semantic intent. Result: commit messages don't accurately describe their contents. Cosmetic impact only — content was correct — but the boundary slippage is a recurring hazard on multi-commit plans.
- **Decision:** Add a `### Commit-boundary sequencing` subsection under the existing `## File verification and git commits` section of workspace `CLAUDE.md`. Rule: when a plan specifies multiple commits and a single file carries changes for more than one commit group, sequence the edits — apply edits for commit N, stage + commit, then apply edits for commit N+1 — rather than editing the file once with all changes and relying on `git add` to split them.
- **Rationale:** Clean commit boundaries aren't ornamental — they're load-bearing for `git bisect`, `git revert`, and code-review clarity. `git add -p` can split by hunk but requires interactive invocation that isn't available in non-interactive agent flows. Edit-then-commit-then-edit produces the same clean boundaries without interactive tooling. The marginal cost is an extra file-state flip per multi-commit plan; the saved cost is not needing retroactive `git reset --soft` + re-commit dances (which are destructive on unpushed commits and blocked on pushed ones).
- **Alternatives considered:** (a) Use `git add -p` — rejected, not viable in agent flows. (b) Rewrite history retroactively via `git reset --soft` when slippage is detected — rejected, destructive for pushed commits and costly for unpushed ones; better to prevent than correct. (c) Bundle all multi-file changes into a single "batch" commit and accept the loss of bisect granularity — rejected, defeats the point of planning commit boundaries. (d) Update commit messages retroactively via `--amend` or rebase — works for pushed commits too via `--force-with-lease` but reviewer's cost-benefit for the cosmetic slippage we already had was "not worth the risk"; the going-forward rule eliminates the need.
- **Follow-up:** The rule is in place for future multi-commit plans. The current session's buy-side commit `dc2c160` carries the slippage (Model Selection in R8+R1 commit rather than R4 commit). Left as-is per operator direction; acts as the motivating example in the decision record.
